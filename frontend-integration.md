## 1️⃣ Global Documentation  

| Item | Value |
|------|-------|
| **Base URL** | `https://api.madtsanda.com` (configured in `config/app.ts` – `APP_URL`) |
| **API version** | `v1` – all routes are prefixed with `/api/v1` (see `start/routes.ts`) |
| **Authentication mechanism** | **JWT** – issued by `AuthService.login` and validated by `AuthMiddleware` (Bearer token in `Authorization` header). |
| **Session / token flow** | 1. `POST /api/v1/auth/login` → returns `{ token, refreshToken }`  <br>2. Token sent as `Authorization: Bearer <jwt>` on protected routes. <br>3. `POST /api/v1/auth/refresh` (if implemented) can exchange refresh token for a new JWT. |
| **Common response envelope** | ```json { "data": <payload>, "meta": { "pagination": … }, "error": null } ```  (see `ForceJsonResponseMiddleware`). |
| **Common error format** | ```json { "error": { "code": <numeric>, "message": <string>, "details": <any> } } ``` (generated by `Handler` exception handler). |
| **Rate limiting** | Not implemented in the current code base. |
| **File upload** | Handled by `FileStorageService` – expects multipart/form‑data, field name `file`. Returns a public URL. |
| **Pagination standard** | Query params `page` (default 1) and `limit` (default 20). Response meta: `{ pagination: { total, perPage, currentPage, lastPage } }`. |
| **Sorting / filtering** | Controllers accept optional `sortBy`, `order` (`asc|desc`) and arbitrary filter query strings – not typed, therefore marked **unknown**. |
| **Error handling strategy** | Centralised in `app/exceptions/handler.ts`. All uncaught errors are transformed to the common error format, HTTP status derived from the exception type. |
| **Validation strategy** | Zod schemas defined in `app/validators/*.ts`. Controllers call `request.validate(ValidatorClass)`. Validation errors are turned into `422 Unprocessable Entity` responses. |
| **Database abstraction** | Lucid ORM (AdonisJS) – models in `app/models/*.ts`. Relationships are defined with `hasMany`, `belongsTo`, etc. |
| **Auth strategy** | JWT + optional role‑based guard (`AuthMiddleware` reads `user.role` from the token). |
| **Architecture pattern** | **Feature‑by‑folder** (controllers, services, transformers, validators, models). Controllers are thin, delegating business logic to services; transformers shape the response payload. |

### `POST /api/v1/auth/login`

**Description** – Authenticate a user and issue a JWT.  

**Headers**  

| Key | Type | Required | Description |
|-----|------|----------|-------------|
| `Content-Type` | `application/json` | ✅ | Must be JSON. |

**Request Body** (validated by `app/validators/auth.ts` → `LoginValidator`)  

| Field | Type | Required | Validation | Description |
|-------|------|----------|------------|-------------|
| `email` | `string` | ✅ | `z.string().email()` | User e‑mail address. |
| `password` | `string` | ✅ | `z.string().min(6)` | Plain‑text password. |

**Example Request**  

```json
{
  "email": "john.doe@example.com",
  "password": "s3cr3t!"
}
```

**Success Response (200)** – Shaped by `AuthService.login` → `UserTransformer`  

```json
{
  "data": {
    "user": {
      "id": "1",
      "email": "john.doe@example.com",
      "name": "John Doe",
      "role": "teacher"
    },
    "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6..."
  },
  "meta": null,
  "error": null
}
```

**Error Responses**  

| Status | Description |
|--------|-------------|
| 401 | Invalid credentials. |
| 422 | Validation error (missing/invalid fields). |
| 500 | Internal server error. |

## 7️⃣ Improvement Recommendations  

| Area | Observation | Recommendation |
|------|-------------|----------------|
| **Role checks** | Role is read from JWT but the middleware only checks existence, not specific permissions per route. | Create a **policy/ability** layer (e.g., using `@adonisjs/acl`) and annotate each route with required abilities. |
| **Error handling** | All errors are wrapped by `Handler`, but custom domain errors (e.g., `EntityNotFound`) are not typed, leading to generic 500 responses. | Define custom exception classes with explicit HTTP status codes (404, 403, etc.) and throw them from services. |
| **Pagination** | Controllers manually read `page`/`limit`. | Extract pagination logic into a reusable **Paginator** service or use Adonis built‑in pagination (`Model.query().paginate(page, limit)`). |
| **File upload validation** | Validators use `z.instanceof(File)` but size limits are hard‑coded in the validator. | Centralise file‑size & mime‑type validation in `FileStorageService` and expose constants. |
| **Duplicate response shapes** | Many endpoints return `{ data, meta, error }` – good, but some controllers forget to wrap the payload. | Enforce a **response helper** (e.g., `ResponseFormatter.success(payload, meta?)`) across all controllers. |
| **Missing rate limiting** | No protection against brute‑force login. | Add `RateLimiterMiddleware` on auth routes (e.g., 5 attempts per minute). |
| **OpenAPI spec** | Documentation is manually generated. | Use `swagger` or `openapi` generator (e.g., `adonis-swagger`) to keep docs in sync automatically. |
| **Testing** | No unit/integration tests for controllers. | Add Jest tests covering each endpoint, especially auth & validation paths. |
| **Enum consistency** | Role strings are duplicated (`'admin'`, `'teacher'`, `'student'`). | Export a shared `enum Role { Admin = 'admin', Teacher = 'teacher', Student = 'student' }` and reuse in validators, services, and DB migrations. |
